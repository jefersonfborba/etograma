## Story
**As a** pesquisador de campo,
**I want** que o app me peça uma pasta de backup ao criar ou abrir um projeto,
**so that** todos os registros sejam salvos automaticamente em CSV incremental sem depender de ação manual, reduzindo o risco de perda de dados em campo.

## Acceptance Criteria

1. Ao criar um novo projeto, o modal exibe um seletor de pasta obrigatório — não é possível salvar sem selecionar uma pasta
2. O handle da pasta é associado ao projeto e persistido no IndexedDB (`config` store, chave `backupDirHandle_{projeto_id}`)
3. Ao abrir um projeto existente sem pasta configurada (migração), um modal de migração é exibido solicitando seleção da pasta antes de liberar o uso do projeto
4. No modal "Editar Projeto", é exibido o nome da pasta atual (ou "Não configurado") e um botão "Alterar pasta"
5. Ao trocar de pasta via "Editar Projeto" com registros existentes: exporta um CSV completo com todos os registros do projeto para a nova pasta e continua o backup incremental a partir daí
6. Ao trocar de pasta via "Editar Projeto" sem registros: apenas troca o handle
7. Cada `addRegistro()` continua fazendo append incremental na pasta do projeto ativo (comportamento existente preservado)
8. O badge `💾 Ativar backup CSV` é removido do header — a ativação passa a ocorrer pelo fluxo de projeto
9. O app funciona normalmente em browsers sem suporte à File System Access API (Chrome/Edge apenas) — em outros browsers, o seletor de pasta não aparece e o backup é silenciosamente desabilitado
10. Os dados do `etograma-dev.html` são integrados ao `etograma.html` — passa a existir apenas um arquivo de produção

A lógica de backup está implementada e funcional em `etograma-dev.html`. A Story 1.5 é principalmente uma **refatoração de UX** — migra o ponto de entrada do backup de um badge no header para os modais de ciclo de vida do projeto.

**Funções já existentes (reutilizar sem modificar):**
- `appendToBackupCSV(r)` — append de linha no CSV da pasta ativa (linhas 1902–1916)
- `backupSuportado()` — detecta File System Access API (linha 1883)
- `nomeFichaBackup()` — nome do arquivo: `etograma_backup_{nome-projeto}.csv` (linha 1885)
- `gerarLinhaCsvBackup(r)` — formata linha CSV com 13 colunas (linha 1889)

**Variáveis de estado a manter:**


**Chave IndexedDB — mudança de design:**

| Versão atual (dev) | Versão 1.5 |
|---|---|
| `backupDirHandle` (global) | `backupDirHandle_{projeto_id}` (por projeto) |

A troca garante que ao trocar de projeto ativo, o handle correto é carregado.

### Exportação completa para nova pasta (AC: 5)

Ao trocar pasta via "Editar Projeto" com registros existentes, usar a função de exportação já existente para gerar o CSV completo:

```javascript
async function exportarCSVParaNovaPasta(projetoId, novoDirHandle) {
  const registros = await registrosDoProjeto(projetoId);
  if (registros.length === 0) return;
  const cab = ["Observador","Projeto","Animal",...].join(";");
  const linhas = registros.map(gerarLinhaCsvBackup);
  const conteudo = "﻿" + cab + "\n" + linhas.join("\n") + "\n";
  const fileHandle = await novoDirHandle.getFileHandle(nomeFichaBackup(), { create: true });
  const writable = await fileHandle.createWritable();
  await writable.write(conteudo);
  await writable.close();
}
```

### Suporte à API por plataforma

| Contexto | File System Access API | Comportamento |
|---|---|---|
| Chrome/Edge (desktop) | ✅ Suportado | Fluxo completo |
| Chrome Android | ✅ Parcial | `showDirectoryPicker` pode não estar disponível — testar |
| Firefox | ❌ Não suportado | Backup silenciosamente desabilitado |
| Capacitor WebView (Android) | ❓ A verificar | Provavelmente não suportado — Story futura com `@capacitor/filesystem` |

> **Nota para @dev:** Em Capacitor/Android WebView, `showDirectoryPicker` provavelmente não funciona. O `backupSuportado()` deve retornar `false` nesses casos, e a Story 1.5 se aplica apenas ao PWA web. Uma story futura pode implementar backup via `@capacitor/filesystem` para o app nativo.
