## Story
**As a** pesquisador de campo usando o app instalado no Android,
**I want** poder configurar a pasta de destino do backup CSV (Documents ou Downloads, com subpasta opcional),
**so that** os arquivos de backup fiquem organizados no local que eu preferir no meu dispositivo, sem depender da pasta padrão Documents.

## Acceptance Criteria

1. Uma opção "Pasta de backup" aparece nas Configurações (painel ⚙️) somente quando `backupCapacitorAtivo()` é `true` (contexto nativo Android)
2. O usuário pode escolher entre dois diretórios raiz: **Documents** (padrão atual) e **Downloads**
3. O usuário pode opcionalmente digitar um nome de subpasta (ex: `pesquisa-2026`) — o CSV será salvo em `{diretório-raiz}/{subpasta}/etograma_backup_{projeto}.csv`
4. Se a subpasta não existir, ela é criada automaticamente na primeira escrita (via `Filesystem.mkdir` com `recursive: true`)
5. A configuração é persistida via `salvarConfig('backupDirNativo', {dir, subpasta})` e sobrevive a reinicializações do app
6. A função `appendToBackupCSVNativo()` usa a configuração salva — lê via `lerConfig('backupDirNativo')` antes de escrever
7. A função `compartilharCSVNativo()` usa o mesmo caminho configurado para localizar o arquivo antes de compartilhar
8. Se nenhuma configuração estiver salva, o comportamento padrão é `Documents` sem subpasta (compatibilidade retroativa com Story 1.6)
9. Uma indicação do caminho atual ("💾 Documents/pesquisa-2026") aparece no indicador visual da home (substitui o texto fixo "💾 Backup automático: Documents")
10. No contexto PWA desktop, a opção de configuração de pasta nativa NÃO aparece (contextos mutuamente exclusivos mantidos)

## Dev Notes

### Arquivo principal
`etograma.html` — single-file PWA, ~2900 linhas

### Funções existentes a modificar
| Função | Modificação |
|--------|-------------|
| `appendToBackupCSVNativo(r)` | Usar `resolverPathBackupNativo()` em vez de `nomeFichaBackup()` + `'DOCUMENTS'` |
| `compartilharCSVNativo()` | Idem — usar path configurado |
| `entrarProjeto()` | Ler config e atualizar texto do `#backup-nativo-info` |

### Diretórios Capacitor disponíveis (sem novo plugin)
| Constante string | Caminho Android | Equivalente |
|------------------|-----------------|-------------|
| `'DOCUMENTS'` | `/sdcard/Documents/` | `Directory.Documents` |
| `'EXTERNAL_STORAGE'` | `/sdcard/` | `Directory.ExternalStorage` |

> **Nota:** Downloads no Android fica em `/sdcard/Download/`. Via Capacitor, usa-se `Directory.ExternalStorage` com path `Download/{arquivo}`.

### Pattern de path com subpasta
```javascript
// Com subpasta: path = "pesquisa-2026/etograma_backup_Projeto.csv", directory = 'DOCUMENTS'
// Sem subpasta: path = "etograma_backup_Projeto.csv", directory = 'DOCUMENTS'
// Downloads:    path = "Download/etograma_backup_Projeto.csv", directory = 'EXTERNAL_STORAGE'
// Downloads + subpasta: path = "Download/pesquisa-2026/etograma_backup_Projeto.csv"
```

### Persistência
- `salvarConfig` / `lerConfig` já existem no projeto (IndexedDB) — reutilizar sem modificação
- Chave: `'backupDirNativo'` → `{dir: string, subpasta: string}`

### Compatibilidade retroativa (AC: 8)
- Se `lerConfig('backupDirNativo')` retorna `null` → usar `{dir: 'DOCUMENTS', subpasta: ''}`
- Garante que instâncias com Story 1.6 instalada continuam funcionando sem configurar nada

### Dependências
- **Depende de:** Story 1.6 (funções `appendToBackupCSVNativo`, `compartilharCSVNativo`, `backupCapacitorAtivo`)
- **Não requer:** Novo plugin Capacitor — usa `@capacitor/filesystem` já instalado
